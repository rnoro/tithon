# DECISIONS (ADR 한 줄 요약)

- ADR-001: 이벤트 seq = messages.msg_seq(AUTOINCREMENT rowid) 재사용 — 별도 카운터 없이 저널 자체가 단조 seq의 진실 공급원.
- ADR-002: 실행 라이프사이클(queued/started/done)을 tithon.* 의사 메시지로 저널에 기록 — 라이브 브로드캐스트와 attach 리플레이가 같은 코드 경로(event_from_message)를 타서 의미 불일치 원천 차단.
- ADR-003: folded 스냅샷은 인메모리 유지 + 실행 done 시 executions.folded_json에 캐시, 데몬 재시작 시 raw 메시지 리플레이로 재구성 — 50k 메시지 실행 중 매 메시지 영속화 비용 회피.
- ADR-004: KernelClient.wait_for_ready 사용 금지 — KernelManager 없는 단독 클라이언트에선 HB 채널 초기 not-beating을 "Kernel died"로 오판. kernel_info 요청 폴링 루프로 대체.
- ADR-005: attach의 무손실 보장은 asyncio 단일 루프 원자성으로 구현 — 구독 등록과 스냅샷/델타 컷오프 계산 사이에 await 없음 + 구독 큐에서 seq<=cutoff 중복 제거.
- ADR-006: v4 검증 셀은 데몬 스레드 기반 증분(x+=1/s) — 커널 셸이 idle로 남아 크래시 직후 x 조회가 즉시 가능, "실행 중 상태 연속성"을 타이밍 의존 없이 검증.
- ADR-007: 커널 ZMQ는 127.0.0.1 TCP(jupyter 표준), 클라이언트 노출은 unix socket(0600) 전용 — TCP 금지 조항은 데몬 바인딩에 적용.
- ADR-008: 이미지 base64는 저널 진입 전에 아티팩트 참조($tithon_artifact)로 치환 — "원본 보존"은 메시지 구조·순서 기준, 페이로드는 파일이 일급(§3.1 두 요구의 양립).
- ADR-009: attach에 last_seen_seq=-1(라이브 전용) 확장 — run CLI가 스냅샷 비용 없이 자기 실행만 추적.
- ADR-010: Widget State Mirror(§3.3) — iopub의 comm_open(target=jupyter.widget)/comm_msg(method update·echo_update)/comm_close를 해석해 widget-state+json 스냅샷을 상시 유지. 바이너리 버퍼는 JSON state에 넣지 않고 buffer_paths별 base64 항목으로 분리 보관(스키마 그대로 → html-manager put_buffers가 재주입). 재접속 비용 = update 횟수가 아니라 최종 상태 크기(tqdm 5만 update → 스냅샷은 막대 1개). 데몬 재시작 시 저널의 comm 메시지(_buffers_b64) 리플레이로 미러 재구성.
- ADR-011: percent 라운드트립은 "구성에 의한 무손실" — parse가 입력을 물리 라인(정확한 종결자 보존) 단위로 분할해 모든 라인을 정확히 한 셀에 귀속시키고 serialize는 그대로 이어붙임. 마커 탐지는 문자열/괄호 인식형(삼중따옴표 문자열·열린 괄호 안의 `# %%`는 마커 아님)이며, 이는 셀 경계(의미)에만 영향, 바이트 충실도와 무관 → 임의 입력에도 0바이트 diff 보장.
- ADR-012: v5의 @vscode/test-electron 통합은 본 환경에서 실행 불가(디스플레이/xvfb-run 부재). stage-b.md 단서 조항에 따라 대체 검증으로 인정 — html-manager가 미러 스냅샷을 jsdom에서 실제 ProgressView DOM(value==max==total, width 100%)으로 렌더(렌더러 에러 없음) + §3.3 폴백(최종 상태 텍스트) 경로 검증. 동일 렌더 로직이 VSCode 렌더러 엔트리(widgetRendererEntry)에 그대로 쓰임. (폴백 전환 근거 기록 조항 충족)
- ADR-013: html-manager의 loadClass는 번들 base/controls와 CSS를 webpack식 require()로 로드 → ESM/jsdom(및 §3.3이 지목한 "렌더러 샌드박스" 제약)과 비호환. HTMLManager를 서브클래스해 @jupyter-widgets/base|controls를 ESM import로 해소(렌더 로직 불변), 그리고 AMD/CDN libembed 경로(__webpack_public_path__) 회피를 위해 HTMLManager를 lib 서브모듈에서 직접 import.
- ADR-014: cell_hash는 데몬이 제출 코드에서 직접 산출(sha256(code))해 저널·스냅샷에 기록 — 클라이언트가 보낸 origin.cell_hash에 의존하지 않아 CLI run(오리진 없음)도 부착키를 가지며, extension의 computeCellHash(sha256 utf8)와 동일 바이트를 해시하므로 일치. origin{uri,range}는 클라이언트 제공분을 그대로 저장(executions.cell_origin_uri/cell_range), 스냅샷 executions[]에 cell_hash·origin 노출 → §3.2 "출력↔셀 매핑"이 두 뷰(텍스트/Cell View)에서 동일 진실로 연결. 구 저널 호환은 additive 마이그레이션(컬럼 추가)로 처리.
- ADR-017: 라이브 출력 동기화 + 렌더 비용 상한(⑩) — 재접속 1회 복원을 넘어 실행 *중* 출력을 셀에 실시간 반영. 나이브한 "이벤트마다 replaceOutput"은 5만 반복 루프에서 렌더러를 마비시키므로, `liveSync.ts`의 `LiveOutputSync`가 (a) throttle/coalesce: 이벤트는 fold만 하고 주입된 Scheduler(프로덕션 50ms ThrottleScheduler)가 윈도우당 1회만 flush — fold가 conflation이라 5만 이벤트→1 sink 호출(단위테스트로 단언). (b) run-merge: 윈도우 내 연속 동일 stream delta는 1 append로 합치되 execute_result/error와의 순서는 보존. (c) delta-append: stream은 증가분만 append(전체 버퍼 재전송 금지) — `\r` 접힘은 VSCode stdout 렌더러에 위임(터미널 의미론). (d) dirty-set: 바뀐 셀만. (e) cell_hash→index는 문서당 1회 메모이즈(이벤트마다 sha256 금지). execId→cell 매핑은 snapshot seed + queued 이벤트의 code 해시. 렌더러 무관(CellSink 인터페이스) → 헤드리스 단위검증(coalescing 상한·순서·\r·clear) + 실 VSCode(v10): startLive가 NotebookController proxy execution으로 셀에 라이브 append, 긴 루프가 셀에서 자라는 것을 observed-growth로 단언.
- ADR-018: 데몬 백프레셔(호스트 보호, ⑨) — 느린/멈춘 클라이언트가 GPU 호스트 메모리를 무한 증가시키지 않게: (1) 구독자별 이벤트 큐 상한(asyncio.Queue maxsize, 초과 시 클라이언트 drop), (2) 전송 stall 타임아웃(SEND_TIMEOUT 내 못 받으면 drop), (3) 연결별 전송 버퍼 상한(websockets write_limit → asyncio transport 버퍼 bound; 실측 50MB 전송에도 transport <100KB 유지), (4) 연결 SO_SNDBUF bound(방어적). drop된 클라는 재접속해 snapshot+delta로 저렴하게 따라잡음(folding 덕). 미전달 바이트는 OS 소켓 버퍼(커널, OS가 상한)에 머물고 데몬 프로세스 메모리는 bound. 상한/타임아웃은 env로 조정 가능(검증 결정성용, 기본값=프로덕션). 검증: drop 경로(큐 overflow + 진짜 hang하는 fake ws의 send-stall)는 pytest test_backpressure.py로 결정론적 단언; 실데몬 호스트 건강성(frozen 클라 붙은 채 빠른 20k run 완료 + status 응답 + 데몬 생존)은 v9. (AF_UNIX 함정: 이 커널은 SO_RCVBUF/SO_SNDBUF로 소켓 버퍼를 줄여도 실효 없고 ~31MB까지 커널이 버퍼링 → 실소켓 drop 유도는 환경의존적이라 단위테스트로 분리.)
- ADR-016: 실 VSCode 통합 검증 복원(⑧, ADR-012 환경 전제 폐기) — ADR-012는 "디스플레이/xvfb 부재"로 @vscode/test-electron 구동 불가라 했으나, 본 환경은 root+apt+네트워크가 가능해 전제가 틀렸다. xvfb + electron 런타임 libs(libgtk-3-0/libgbm1 등)를 설치하고 @vscode/test-electron으로 실제 VSCode(1.124.2)를 xvfb 아래 띄워 검증한다(v8.sh). in-host mocha 테스트가 fixture .py를 tithon-py 노트북으로 열고 Tithon 컨트롤러를 커널로 select(notebook.selectKernel{id:"tithon",extension:ext.id} — @types/vscode 1.120엔 NotebookEdit.updateCellOutputs가 없어 출력 설정은 createNotebookCellExecution().replaceOutput 경로가 유일 → 컨트롤러 select 필수)한 뒤 tithon.restoreOutputs를 실행, 셀에 folded 출력(stdout "0\n1\n2\n"/execute_result 42/error ValueError)이 실제로 쓰였음을 NotebookCell.outputs로 단언. 데몬 시드는 검증된 SessionClient(seed.ts)로 cellSource를 제출해 cell_hash 일치 보장. v8은 네트워크(VSCode 다운로드)+xvfb 의존이라 hermetic한 `make verify`(v1~v7)에서 분리하고 `make verify-d`로 제공. → sessionController는 더 이상 "스파이크"가 아니라 실 VSCode에서 구동 검증됨.
- ADR-015: 재접속 출력 복원(클라이언트 구독 절반, Phase 1 ⑦) — 데몬은 이미 완성됐고 빠진 것은 "스트림 구독→folding→복원→셀 부착" 클라이언트 경로였다. (a) `outputFold.ts`는 folding.py의 정확한 TS 포트(\r/\n/\b StreamBuf, clear_output(wait), update_display_data, stream/display_data/execute_result/error). 라이브/델타는 raw 이벤트가 오므로 클라이언트가 직접 fold해야 하고, `seed()`로 스냅샷의 이미 folded된 outputs에서 이어 fold(진행 중 실행도 일관). (b) `sessionClient.ts`가 attach(last_seen_seq)로 snapshot+delta+live를 소비해 실행별 fold 상태 유지 → `restoreInto(cells)`가 cellAttach로 셀 부착. cell_hash는 snapshot에만 실리므로(queued 이벤트엔 없음) 실행 완료 후 fresh attach(0)가 정상 복원 경로 — 이게 곧 "탭 닫고 tunnel로 재접속" 시나리오. (c) `sessionController.ts`는 NotebookController.replaceOutput로 복원 출력을 셀에 쓰는 VSCode 바인딩(스파이크, 실 VSCode 미구동 — ADR-012 기조). 검증 v7: 실제 데몬+커널에 셀 제출→실행→fresh SessionClient 재접속이 folded 출력을 복원·부착(stream "0\n1\n2\n"/execute_result 42/error ValueError), 그리고 라이브 raw fold == 데몬 snapshot fold 동등성. restore.test.ts는 소켓 없으면 skip(npm test 격리 유지)이나 v7.sh는 데몬 가동 중 skip을 FAIL로 처리(안티치팅: 실검증 무력화 금지).
- ADR-019: 셀 출력↔셀 매핑 견고화 (실 tunnel 사용자 버그 #1/#2 수정) — 실제 VSCode tunnel 사용에서 두 버그 확인. #1: 셀 실행 버튼을 눌러도 출력이 안 보이고 tithon.restoreOutputs를 따로 실행해야 보임 — DaemonClient.execute()가 ack 직후 소켓을 close하므로(데몬 로그: attach→execute→CLOSE→output) 출력이 도착하기 전에 연결이 끊겨 출력을 받을 구독자가 없었다. 네이티브 play 버튼의 executeHandler는 () => undefined(무동작)였다. → executeHandler/CodeLens가 제출 전 ensureLive(notebook)로 영속 구독자를 먼저 attach(실 Extension Host v11로 검증: 수동 live 단계 없이 셀 출력 표시). #2: 여러 셀을 모두 실행하면 마지막 셀 출력만 맨 위 셀에 표시 — restore의 attachOutputs proximity 폴백이 (a) 영속/전역 저널의 hash 불일치 실행들을 (b) range {0,0}(origin 없음) 또는 cell-index range(line range가 아님)로 비교해 전부 cell 0에 collapse(마지막이 win)시킨 것. attachOutputs를 **정확 cell_hash 일치 전용**으로 바꿔 불일치 실행은 cell에 붙이지 않고 skip(proximity는 동일 hash 중복 셀 tie-break로만 잔존), restoreInto(cells, fileUri)로 **현재 파일 uri로 스코프**(전역 저널의 타 파일 실행 차단), executeHandler의 origin.range를 cell-index가 아니라 **line range**로 통일(CodeLens·docCellsFromParsed와 동일 좌표). 트레이드오프: 실행 후 셀을 편집하면 옛 출력은 복원 안 됨(엉뚱한 셀에 붙느니 안 붙는 게 낫다). 검증: cellAttach.test.ts에 collapse 회귀(불일치 3개 → 0개 부착) + skip 단언 추가(기존 proximity-stale 테스트는 새 계약으로 교체, 본 ADR에 사유 기록), 실 VSCode v12(blank-line 포함 멀티셀 각자 셀에 매핑)·v11(네이티브 실행). 더 근본적으로 cell_hash 매핑은 공백 1바이트만 달라도 stale → 향후 라인-앵커/안정 ID 도입 여지(백로그).
- ADR-020: Cell View로 추가한 셀의 직렬화 glue 버그 수정 (#3) — 사용자가 +Code 버튼으로 만든 셀을 저장하면 `print("Hello")# %%`처럼 다음 셀 마커가 앞 코드줄 끝에 붙어 나왔다. 원인: notebookSerializer의 synthesizeCell이 마지막 body 줄의 terminator를 ""로 둬서(끝 개행 없음) serialize가 다음 `# %%`를 바로 이어붙임 → 그 마커는 줄 시작이 아니므로 parse가 마커로 인식 못해 파일이 셀 1개로 붕괴(3셀→1셀, 재현·단언). 수정: serializer.ts에 순수 헬퍼 bodyLinesFromText(value)를 추가해 모든 줄(마지막 포함)을 "\n"으로 종결(입력의 끝 개행은 정확히 1개로 정규화)하고 synthesizeCell이 이를 사용. 기존 셀(메타데이터 stored)은 여전히 verbatim 라운드트립(ADR-011, v6 0바이트 diff 불변). 회귀: serializer.test.ts에 glue 금지(/[^\n]# %%/ 부재) + 3셀 라운드트립 + 끝개행 정규화 단언. (참고: 편집된 기존 셀이 stored 메타데이터로 옛 내용을 직렬화하는 별개 latent 버그는 백로그.)
- ADR-021: 출력→셀 매핑을 디스크 파일이 아니라 **메모리 노트북 셀**에서 산출 — 실 tunnel 사용자가 0.0.2 설치 후에도 출력이 안 떴는데, 원인은 startLive/restore가 `notebook.uri`의 디스크 .py를 다시 parse해서 hashIndex를 만든 것. 사용자의 test.py는 옛 확장의 glue 버그로 1셀(붕괴)이라 parse 해시가 실행 셀(`print("hello")`)과 불일치 → execId 미매핑 → 영속 구독자가 붙어 있어도 출력 드롭(재현·확인). 더 일반적으로 디스크와 노트북이 다르면(저장 안 한 편집, 손상 파일) 항상 깨졌다. 데몬은 cell_hash=sha256(제출 코드)=sha256(cell.document.getText())로 저널하므로, 라이브/복원 인덱스도 **열린 노트북의 getText()를 그대로 해시**해야 일치한다. `cellsFromNotebook(notebook)`가 각 셀을 body 1줄(text=getText(), terminator="")로 만들어 cellSource===getText() → computeCellHash가 데몬 해시와 정확히 일치. startLive·restore가 이 함수를 사용(디스크 read 제거). markup 셀도 포함해 인덱스가 notebook.cellAt(i)와 정렬. 검증 v13: 디스크는 DISKVERSION인데 메모리에서 셀을 EDITED로 고치고(노트북 dirty) 실행→셀에 EDITED 출력(스테일 디스크 무시) 단언. → 저장 안 한 편집·손상 파일에도 출력 매핑이 견고. (executeHandler의 origin.range용 cellLineRanges는 디스크 parse 유지하나 range는 동일-해시 tie-break 전용이라 무해.)
- ADR-022: 라이브 셀-해시 인덱스를 셀 추가/편집 시 갱신 — 0.0.3에서도 사용자가 "처음 셀은 출력 나오는데, 셀 추가 후 실행하면 안 나오고 Restore를 해야 보인다"고 보고. 원인: LiveOutputSync의 hashIndex가 startLive 시점에 한 번만 구성돼, 그 후 추가된 셀의 cell_hash가 인덱스에 없어 queued 이벤트가 execId→cell 매핑에 실패→출력 드롭(restore는 매번 현재 셀을 다시 읽으니 됐던 것). 수정: `LiveOutputSync.refreshCells(cells)`로 인덱스 재구성, startLive가 `onDidChangeNotebookDocument`를 구독해 셀 변경 시 refresh하고, executeHandler/CodeLens도 제출 직전 `refreshLive(notebook)` 호출(이벤트 순서 레이스 제거). 검증: liveSync.test 단위(인덱스 갱신 전 드롭→후 매핑) + 실 VSCode v14(셀0 실행으로 라이브 시작→셀 추가→그 셀 실행이 restore 없이 라이브 출력). verify-d=v8 v10 v11 v12 v13 v14. (디버깅 중 발견한 테스트 함정: notebook.cell.execute는 ranges 인자보다 에디터 selection을 따르므로 비활성 셀 실행엔 selection 먼저 설정해야 함.)
- ADR-023: 중간 재접속 = 이전 출력 복원 + 실시간 계속 (라이브 경로에 스냅샷 렌더 추가) — 사용자가 "학습 루프처럼 긴 셀 실행 중 재접속하면 이전 출력 복원 + 이후 실시간 출력이 끊김없이 보여야 한다(처음부터 보고 있던 것처럼)"고 요구. 검증(v15)으로 기존 startLive는 **실시간만** 렌더하고 재접속 이전 출력(0~5)을 빠뜨림을 확인 — daemon/client는 snapshot+delta를 지원하나 VSCode 렌더 절반이 스냅샷을 안 그렸다. 수정: startLive가 attach(0) 직후(동기, onEvent 와이어링 전 — 이벤트 유실/중복 없는 창) 각 매핑된 exec의 folded 출력을 `VSCodeCellSink.seedCell(idx, items, terminal)`로 셀에 렌더하고 stream 블록을 streamOut에 등록 → 이후 라이브 delta가 **같은 stdout 블록에 이어붙어** 하나의 연속 스트림이 됨(실행 중 exec는 running 유지, 완료 exec는 done 처리). cutoff 경계로 스냅샷(≤cutoff)과 라이브 delta(>cutoff)가 겹치지 않아 중복 없음. 검증 실 VSCode v15(루프 ~4줄 후 재접속→셀에 0~29 전체) + 스크린샷(연속 렌더 육안 확인). verify-d=v8 v10 v11 v12 v13 v14 v15.
- ADR-024: 렌더 스크린샷 검증 표준화 — "출력이 화면에 실제로 그려지는지"를 데이터 모델(cell.outputs)뿐 아니라 픽셀로 확인하라는 사용자 요구. `verify/shot.sh <suite>`가 실 VSCode 통합 스위트를 스크린샷 가능한 Xvfb에서 구동하고 HOLD 창 동안 여러 프레임을 캡처해 가장 렌더된 프레임을 보관, 픽셀 std-dev>0.02로 blank-screen을 자동 차단(PNG는 verify/shots/에, gitignore). `make shot`(reconnect+screenshot). 한계: 텍스트 OCR 단언은 안 함(육안+모델 단언 병행), 실제 tunnel 원격 왕복·사람 손 클릭은 미커버(사용자 실기기가 최종 관문).
